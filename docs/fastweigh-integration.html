FastWeigh Integration — Architecture & Implementation Document

The §§ 1-2 webhook spec is still current. §§ 3-5 contain large obsolete sections (LPN-linked v1/v2 model, SW_OverweightPermit table, browser-based measurement plan) — those are marked SUPERSEDED below with pointers to the authoritative source in architecture.md and decisions.md. § 6 implementation status is partially stale — updated below. When this doc and architecture.md disagree, architecture.md and the ADRs in decisions.md win.

ALSO SUPERSEDED 2026-06-02: Any references in §§ 1-2 or implementation status to transactionGuid / ticketPending GraphQL / Tier 1/2/3 framing as the correlation source. Current model: tare↔loadout pairs on a configurable PosType19 tare-ticket field (default truck_id), provider-agnostic across Apex XML and FastWeigh JSON. 1. Overview

FastWeigh is being added as a second cloud-based ticket data provider alongside the existing Command Alkon APEX local integration. This establishes the cloud ticket ingestion pattern that Command Alkon's upcoming "CommandCloud" platform will also use.

Provider architecture:

Provider Layer (webhook receivers)
  +-- FastWeigh  (Svix webhooks, JSON) <-- v1 implemented + E2E tested
  +-- CommandCloud (future, likely webhooks)
  +-- APEX (existing, local HTML -- unchanged)
        |
        v
  Pass-Through Layer (strip routing fields, add provider tag, forward raw JSON)
        |
        v
  Common Pipeline (WebSocket push -> NVR CloudTicket adapter -> transaction processing)

Design decision (2026-05-21): No normalization layer. Cloud tickets pass through as raw JSON — the NVR backend only needs ticket (number) and uniqueid for processing. All other fields are provider-specific key-value pairs stored and displayed as-is. The CloudTicket adapter on the NVR wraps the dict to provide the same interface as APEX Ticket objects (.rgsOverlay, .bTicket, .bUniqueID, etc.).

2. Core Webhook Flow (v1 — Implemented)

FastWeigh (Svix)
  |
  | POST /sw/webhooks/fastweigh (HMAC-SHA256 signed)
  v
CloudAPI
  | 1. Verify Svix signature (raw body + svix-id/timestamp/signature headers)
  | 2. Idempotency check (deduplicate by svix-id in SW_WebhookLog)
  | 3. Strip routing fields, add provider tag (raw JSON pass-through)
  | 4. Lookup NVR WebSocket by serial from ticket payload
  v
WebSocket push (ticket_event message)
  |
  v
NVR (on-premise)
  | 1. ThreadCloudWebsocket receives ticket_event
  | 2. Creates Ticket_Received event (ticket data is a raw dict)
  | 3. Dispatches to ThreadTruckTransaction via addEvent()
  | 4. _saveTicket detects dict, wraps in CloudTicket adapter
  | 5. Transaction processes: captures images, fetches measurement from spot-decoder
  | 6. Uploads completed transaction back to CloudAPI via POST /sw/
  v
CloudAPI stores transaction -> visible in ScaleWatcher Cloud portal

NVR offline fallback: If the NVR WebSocket is disconnected, CloudAPI creates a fallback SW_Transaction with sTag='cloud-only' and logs to SW_WebhookLog with status='fallback'. Still returns 200 to Svix to avoid retries.

3. Tare vs Loaded Ticket Flow — SUPERSEDED

SUPERSEDED 2026-05-22. The original content of this section assumed LPR (license-plate reading) was the cross-scale correlation mechanism for linking tare → loaded transactions. LPR is OFF the table per customer decision — the customer says LPR is "not stable" at their sites, and the production approach uses other means (QR-permit-driven, FastWeigh-supplied correlators, or operator entry). The v1 "independent transactions linked by sLpn" and v2 "CloudAPI auto-linking by LPN" models in the original draft are both invalid.

Current authoritative source for transaction lifecycle: architecture.md §§ 0c and 2 (NVR fire-and-forget per-truck-batch upload), decisions.md ADR-005 (raw JSON pass-through), and the forthcoming transaction-lifecycle-deep-dive.md (in progress) which will detail single-scale vs dual-scale handling, the paired-main-transaction model, and the correlation-key options once FastWeigh credentials confirm what's in the payload.

The legitimately-open question remains: how FastWeigh's webhook payload differentiates tare vs loaded (a type field, ticket-number convention, separate event types, or custom UDF). That is captured in open-questions.md §1.1 and §2.1 — pending FastWeigh sales call.

(Original §3 content omitted; was scale flow diagrams with Lpr_Read start events and "both transactions share the same LPN" — both contradicted by LPR-off decision.)

4. Overweight Permits — SUPERSEDED (Implemented Differently)

SUPERSEDED 2026-05-22. Every assumption in the original §4 turned out wrong once customer answered (2026-05-21) and we built the real solution. Specifically:

  • NOT a "new feature not yet implemented." Built end-to-end. NVR recorder/src/scalewatcher/permit_client.py (541 lines, 54 unit tests) validates permits against a real FDOT permit (ecda589c-7aaa-4c7b-9771-122133e533fd).
  • No upload-and-store-permit mechanism needed. Driver shows the permit's QR code; the QR decodes to a public FDOT URL (pas.fdot.gov/Public/ViewPermit/{guid}); we download the official PDF directly from FDOT and parse it. See ADR-007.
  • No SW_OverweightPermit table. Permits ride as Permit_Validated events in SW_Event (parsed fields in sData JSON), and the PDF is stored as a transaction image. Same shape as ADR-002 for measurement. See ADR-006. The bPermit FK column on SW_Transaction is a vestige with no FK target.
  • No "associate permits with specific trucks by LPN." LPR is OFF; permits are per-permittee (carrier), not per-vehicle, per customer confirmation 2026-05-21. We don't need to associate at the entity level — we associate the validation event with the transaction it was validated on.
  • No CRUD-routes-for-permit-management. There's nothing to manage at the cloud level — every truck shows the permit on every trip, we validate live, no permit registry to keep current. The permit PDF is the audit trail, stored per-transaction.
  • No portal page for permit management. Phase 6 portal work is still pending but will display permit validation on the transaction-detail view (permit number, expiration, validity, PDF link), not maintain a permit database.
  • The "questions to resolve" listed in the original §4 are all answered — see open-questions.md §1.2 (all [x] items) and ADR-006/ADR-007.

Current authoritative source: architecture.md § 6 (permit subsystem), ADR-006 and ADR-007 in decisions.md, and the customer-context section architecture.md § 0b operational rule 3.

(Original §4 content omitted; described an entity-modeled permit registry with SW_OverweightPermit table keyed by LPN — none of which we built or will build.)

5. Axle Measurement Tool — SUPERSEDED (Implemented in Spot-Decoder C)

SUPERSEDED 2026-05-22. The original §5 proposed a browser-based measurement tool drawing on captured images in the portal — "Option B" in the original open-questions.md §4.1 decision matrix. Decision 2026-05-19 (ADR-003): measurement UI moved to spot-decoder C because browser focus issues are unreliable in a multi-window scale-house environment. The browser approach was rejected.

What's actually built:

  • nvr/spot-decoder/src/measurement.c (838 lines) — Federal Bridge Formula B (23 USC 127 + FL 316.535), normalized 0.0-1.0 coordinate system, calibration state machine with INI persistence, HTTP API on port 88, USB HID key handler. 111/111 bridge formula tests passing.
  • nvr/recorder/src/scalewatcher/measurement_client.py (165 lines) — HTTP client that fetches measurement status + JPEG screenshot from spot-decoder at transaction-finish time. Wired into thread_truck_transaction.py:710-762.
  • Storage: rides as MeasurementRecorded event in SW_Event, summary fields bAxleCount (int) and fBridgePass (tinyint) on SW_Transaction. See ADR-002.
  • Portal display (measurement panel on transaction-detail view) is Phase 6 — not started, see cross-repo-status.md.

Current authoritative source: architecture.md § 5 (the measurement subsystem in full), ADR-002 (no SW_Measurement table), ADR-003 (spot-decoder C, not browser). Hardware track (controller, PCB, enclosure, Arduino firmware) lives in cv-axle-controller — see architecture.md § 8.

(Original §5 content omitted; described a browser-based canvas tool drawing lines on event images — explicitly rejected by ADR-003.)
