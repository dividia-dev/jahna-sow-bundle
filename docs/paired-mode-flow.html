Paired-mode flow

Paired-Mode Flow Reference

How the NVR recorder and CloudAPI handle scale-watcher transactions in every combination of: legacy vs paired code path, single-scale vs two-scale layout, cloud-uploading vs local-only.

This document reflects the code on branch tasks/86b9y6jc5 after commits 7aa042f67 (paired rename) and 172abb29f (durability fix) on the nvr side and 77c8faf (sCorrelationId dedup) on the cloudapi side.

CONFIRMED CURRENT 2026-06-02: the sTareField/sTareValue discriminator + sMatchInbound field described below is the authoritative correlation mechanism for both Apex and FastWeigh ticket sources. Tare↔loadout keys on truck_id by default (configurable). Any older docs in the hub referencing transactionGuid or ticketPending GraphQL as the correlation source are superseded — this file is current.

Configuration switches that select the path

Three switches determine which path a transaction follows:

Switch Where Default Effect
PosType19.fPairTareLoadout DB column on PosType19 row false Master switch. Enables paired-mode for that POS.
/root/.recorder_cloudsw file presence NVR filesystem absent Enables cloud-SW uploader thread. Created by Dividia support, not the operator.
PosType19.bPairTimeout DB column on PosType19 row 480 (minutes) How long a cached tare may sit before the expiry sweeper ships it as an orphan tare transaction.

The "scale layout" — single scale (truck tares AND loads on the same scale) vs two-scale (tare on scale A, loadout on scale B) — is not a config switch. It's a property of how the customer set up their POS in dview (one POSType19 entry per scale). Whether the truck is taring or loading is determined by the ticket data via the sTareField/sTareValue discriminator.

The default values mean: a fresh customer install runs identical to the APEX flow that's been in production for years. Paired-mode only activates if the customer explicitly turns fPairTareLoadout=true on a PosType19, which requires going through the dview setup UI (admin user sees the section only if cloud-SW is enabled).

Recorder flows

Path A — Legacy, no cloud-SW

When: fPairTareLoadout=false, no /root/.recorder_cloudsw file.

This is the original APEX flow that's been running for years on every APEX customer without cloud upload.

Flow:

  1. Truck pulls onto scale, Scale_TruckEntering fires.
  2. ThreadTruckTransaction state machine moves WAITING → PROCESSING.
  3. Events accumulate in truck.cloudEvents[] and images grab into truck.imgs[]: Scale_TruckOn, Lpr_Read (if LPR), Valve_Opened /Valve_Closed (if valve), Ticket_Received (when ticket arrives), Scale_TruckLeaving.
  4. _handleFinishedState fires:
    • cloudAdd dispatcher fires with (bID, cloudEvents, imgs) — but no listener is wired (cloud-SW thread didn't start because the gate file is absent), so the call is a no-op.
    • _classifyPairedPhase returns None (paired is off), so we drop past the paired branch.
    • _saveStandardTicket (or _saveInboundTicket for tare-side tickets where bTicket=='0') writes the local PDF to ticketsdir.
  5. _clearTruck resets state to WAITING.

Result: local PDF on the NVR filesystem. Operator views via dview's local playback search. No cloud upload. No paired cache writes.

Path B — Legacy, with cloud-SW

When: fPairTareLoadout=false, /root/.recorder_cloudsw file present.

This is the APEX cloud-SW flow that customers like Quikrete-Midlothian, Capitol-Aggregates, MMM-sites run today.

Flow:

  1. Same steps 1–3 as Path A.
  2. _handleFinishedState fires:
    • cloudAdd(bID, cloudEvents, imgs) dispatches to ThreadCloudSW.addEvent. That writes a JSON file to clouddir (with a fresh sCorrelationId), queues it for upload.
    • _classifyPairedPhase returns None (paired is off).
    • _saveStandardTicket / _saveInboundTicket writes the local PDF.
  3. ThreadCloudSW.run() loop picks the JSON file off the queue, POSTs to <cloud-api>/sw, deletes the file on 200 response.

Two-scale variant: if the customer has two PosType19 entries on the same NVR (one per scale), each scale cycle is its OWN independent cloudAdd → its OWN SW_Transaction in the cloud. The cloud sees TWO rows per truck visit with no link between them. The legacy ApexInboundCache (sProvider=apex) image-cache is what stitches the tare-side jpgs into the loadout-side LOCAL pdf — but that's filesystem-only, no cloud awareness.

Result: local PDF + one cloud transaction per scale cycle.

Path C — Paired, no cloud-SW

When: fPairTareLoadout=true, no /root/.recorder_cloudsw file.

This is a "what if" combination — not a customer configuration we expect to see in production, but useful to understand what happens.

Flow:

  1. Same steps 1–3 as Path A: events accumulate in RAM.
  2. _handleFinishedState fires:
    • _classifyPairedPhase returns 'tare' (if ticket has bTicket=='0' per default discriminator) OR 'loadout' (otherwise).
    • On 'tare': _cachePairedTareTruck pickles the full TruckTransaction into ApexInboundLog.sPairedState. Function returns STATE_WAITING immediately — cloudAdd is NEVER called, _saveStandardTicket is NEVER called.
    • On 'loadout': _getHeldPairedTare looks up the matching cached tare, merges its cloudEvents + imgs into the loadout truck. Then falls through:
      • cloudAdd fires but has no listener (cloud-SW thread didn't start) — no-op.
      • finalizePaired deletes the cache row.
      • _saveStandardTicket writes the local PDF, which now includes the merged tare-side images.

Result: local PDF on the NVR with both tare and loadout images combined. NO cloud upload (gate file absent). Tare-side scale events are LOST from the local PDF flow's perspective if the cycle is a tare cycle — _saveStandardTicket never runs for tare-only cycles in paired mode, so the tare local PDF that legacy customers got is no longer written.

Why this configuration is unusual: paired-mode was designed to collapse two scale cycles into ONE cloud transaction. Without cloud-SW, the only effect is suppressing the tare-side local PDF. Customers who don't use cloud-SW have no reason to turn paired-mode on, and the dview UI only shows the toggle when cloud-SW is enabled (per ApexDialog.setPairedSectionVisible(checkHasCloudSw())). Operators can't reach this state through the UI without manual DB editing.

Path D — Paired, with cloud-SW (the Jahna case)

When: fPairTareLoadout=true, /root/.recorder_cloudsw file present.

This is the production target for Jahna and the FastWeigh integration.

Flow (tare cycle):

  1. Truck pulls onto tare scale, Scale_TruckEntering fires.
  2. State machine moves WAITING → PROCESSING.
  3. Events accumulate: Scale_TruckOn, Lpr_Read (if LPR), Measurement_Recorded (from spot-decoder if configured), Permit_Validated (if QR scan), Ticket_Received (tare ticket with bTicket=='0'), Scale_TruckLeaving.
  4. _handleFinishedState fires:
    • _classifyPairedPhase returns 'tare'.
    • _cachePairedTareTruck pickles the full TruckTransaction into ApexInboundLog.sPairedState with sProvider='paired'. The events are stored as-is — phase ('tare' vs 'loadout') is implicit at the cloud side from each event's own ticket data (the tare's Ticket_Received has number==sTareValue). No explicit phase tag rides along; the earlier Event.shipmentID overload was removed in commit 9d3d18b43 because it was dead on the wire and conflicted with PSCL's legitimate use of the same field.
    • Returns STATE_WAITING. cloudAdd is NOT called.

Flow (loadout cycle, sometime later — minutes to 8 hours):

  1. Truck pulls onto loadout scale, Scale_TruckEntering fires.
  2. New TruckTransaction accumulates: scale events, Ticket_Received (loadout ticket with bTicket != '0'), Scale_TruckLeaving.
  3. _handleFinishedState fires:
    • _classifyPairedPhase returns 'loadout'.
    • _getHeldPairedTare(truck.ticket) searches ApexInboundLog for a row matching truck.ticket.oXML on the configured rgsMatch fields. Returns (oCachedTare, bTareID, bTareTimestamp) on hit OR None.
    • On hit:
      • truck.merge(oCachedTare) appends the cached tare's cloudEvents/imgs into the loadout truck. No event mutation — see the tare-side note above re: the removed phase-tag dead code.
      • pairedFinalizeKey = (bTareID, bTareTimestamp) for later.
    • On miss:
      • Alert_LoadoutWithoutTare event appended to the loadout's cloudEvents. No merge.
    • cloudAdd(bID, cloudEvents, imgs) dispatches to ThreadCloudSW.addEvent — writes the JSON file to clouddir with a fresh sCorrelationId. This is the durability point.
    • Only AFTER cloudAdd returns: if we had a hit, finalizePaired(bTareID, bTareTimestamp) deletes the cache row.
    • _saveStandardTicket writes the local PDF (merged imgs).
  4. ThreadCloudSW.run() uploads the JSON file, gets back the cloud transaction id, deletes the file.

Result: ONE local PDF, ONE cloud SW_Transaction containing BOTH scale cycles' events in chronological order.

Flow (orphan tare — never paired):

If a tare cycle never gets a matching loadout within bPairTimeout minutes (default 480 = 8 hours), the periodic sweep _processExpiredPaired runs:

  1. getExpiredPaired(bNow) returns rows older than bPairTimeout without deleting them.
  2. For each row: append Alert_TareWithoutLoadout to the truck's cloudEvents, then cloudAdd (writes JSON to clouddir).
  3. Only AFTER cloudAdd: finalizePaired(bID, bTimestamp) deletes the row.

Result: the tare-side data uploads as its own SW_Transaction with the alert appended. Cloud sees a one-cycle transaction with an alert icon. Operators can investigate.

Flow (loadout without matching tare):

If a loadout arrives but no cached tare matches (getHeldPairedTare returns None):

  1. Alert_LoadoutWithoutTare event appended.
  2. Normal cloudAdd for the loadout side only.
  3. No finalizePaired call (no row was matched).

Result: the loadout uploads as a one-cycle transaction with the alert appended. Per Jahna policy this represents a real anomaly to investigate — operator should always tare.

Recorder durability story

The paired-mode bridge step (cache row carrying state between the two scale cycles, possibly hours apart) introduces a new failure mode that the legacy flow doesn't have. Three durability-relevant points:

  1. Cache write is in MariaDB. The pickled TruckTransaction lives in the ApexInboundLog table. Survives recorder restart, OS reboot, power loss.
  2. Cloud upload is in clouddir filesystem first, queue second. ThreadCloudSW.addEvent writes the JSON file to disk BEFORE returning. Then the run loop reads from disk and POSTs. If the recorder crashes after writing the file but before posting, the next boot's _loadEventsFromDisk re-queues the file. Uploads are durable to crashes.
  3. The bridge step's finalize-after-cloudAdd ordering (the fix in commit 172abb29f): the cache row is held across the merge → cloudAdd window so a crash there leaves the cache row in place; the next loadout retry re-uploads, and cloud-side sCorrelationId dedup catches the duplicate.

The combination provides recovery from a recorder crash at any point in the paired flow with at most one in-flight TruckTransaction's worth of loss — the same loss exposure as the legacy APEX flow.

CloudAPI ingest flow

Legacy path (every customer not on paired-mode + every legacy NVR)

The NVR POSTs the JSON payload to <cloud-api>/sw. The payload looks like (abbreviated):

{
  "serial": 4001,
  "scale": 1,
  "timestamp": "2026-05-22T13:45:00Z",
  "tzoffset": -5,
  "sCorrelationId": "abc123...",
  "events": [
    {"timestamp": ..., "name": "ScaleTruckEntering", "data": {...}, "imgs": [...]},
    {"timestamp": ..., "name": "Lpr_Read",           "data": {...}, "imgs": [...]},
    {"timestamp": ..., "name": "ScaleTruckOn",       "data": {...}, "imgs": [...]},
    {"timestamp": ..., "name": "TicketReceived",     "data": {...}, "imgs": [...]},
    {"timestamp": ..., "name": "ScaleTruckLeaving",  "data": {...}, "imgs": [...]}
  ],
  "cams": [1, 2]
}

cloudapi/app/controllers/scalewatcher/transaction.js's create():

  1. Validates the payload (events present, serial valid, etc.).
  2. Reads req.body.sCorrelationId (may be absent on uploads from older recorder firmware — that's fine, null allowed under the UNIQUE index).
  3. If correlation id is non-null and present in SW_Transaction.sCorrelationId: returns the existing transaction id with deduped: true. No new row created.
  4. Otherwise: inserts a new SW_Transaction row carrying the correlation id. If the INSERT races and hits ER_DUP_ENTRY, the pre-check is re-run and the existing id is returned.
  5. Iterates events[], INSERTing each as a SW_Event row pinned to its real timestamp.
  6. Computes alert events (NoTicket, GrossWeightMismatch, etc.) based on scale data + ticket presence.
  7. Saves images to disk via FILE_STORE.

Paired path (Jahna + future paired-mode customers)

Today, after commits 7aa042f67 + 172abb29f + 77c8faf: the paired upload payload looks IDENTICAL to a legacy payload, just with more events in the events[] array (both scale cycles' events chronologically interleaved). The cloudapi side has no concept of "this is paired" — it just renders whatever events arrived.

Specifically:

  • The merged transaction contains TWO ScaleTruckEntering, TWO ScaleTruckOn, TWO ScaleTruckLeaving events.
  • The TicketReceived events may be one or two (tare ticket with bTicket=='0' and the loadout ticket with a real number).
  • The Alert_LoadoutWithoutTare (orphan loadout) or Alert_TareWithoutLoadout (orphan tare) shows up as an AlertLoadoutWithoutTare/AlertTareWithoutLoadout SW_Event row.
  • The portal renders all events along a horizontal timeline by dTimestamp — the tare-cycle icons cluster on the left, the loadout-cycle icons on the right, with a gap in between.

Known portal-side weirdness with this minimal-shape payload:

  • The portal's Load column (ScaleTruckLeaving - ScaleTruckEntering) computes against the merged event list and may compute a misleading number — last leaving minus first entering = whole tare-to-loadout span = hours, not minutes. Per discussion 2026-05-23 we are intentionally NOT addressing this in the current pass; a future cloudapi change will switch to a phase-aware computation.
  • The transaction's gallery view numbered breadcrumb will show 8-12 steps instead of the usual 5-7.

Future shape (not yet implemented): the upload payload will carry a paired.tare sub-document with the tare cycle's data so cloudapi can expand it into phase-tagged SW_Events. Portal can then render tare-phase icons visually distinct from loadout-phase icons, and the Load column computation can become phase-aware. Separate work item.

Idempotency / dedup mechanics

The sCorrelationId column on SW_Transaction:

  • VARCHAR(64) NULL
  • UNIQUE index idx_transaction_correlation_id
  • MariaDB allows multiple NULLs under a UNIQUE index, so legacy uploads (no correlation id sent) coexist freely.

create() flow when the recorder retries (e.g., after a recorder crash between cloudAdd writing the JSON file and finalizePaired deleting the cache row, or any network timeout that causes the recorder to re-queue the same file):

  1. Recorder reads the same JSON from clouddir with the same sCorrelationId.
  2. POST hits cloudapi.
  3. cloudapi pre-checks: SELECT id FROM SW_Transaction WHERE sCorrelationId=? — finds the existing row.
  4. Returns 200 {id, deduped: true} immediately without any further work.
  5. Recorder sees a 200, deletes the JSON file from clouddir.

Race condition: if two retries hit cloudapi simultaneously, the pre-check SELECT might miss the in-flight INSERT. The UNIQUE index backstops this — the second INSERT raises ER_DUP_ENTRY. cloudapi catches that error, re-runs the SELECT, and returns the existing id the same way. Belt-and-suspenders.

Permit PDF delivery (now wired)

Permits flow into the cloud through the standard event+image pipeline, not via a separate path.

Recorder side (thread_truck_transaction.py:795-848):

  1. When a QR-scan event arrives during a transaction with a permit URL, permit_client.lookup_permit(qr_text) fetches the FDOT permit page, downloads the authoritative PDF, parses it for metadata (permit number, status, expiration, GVW, axle count, vehicle config), and validates.
  2. A Permit_Validated event is appended to truck.cloudEvents with the parsed metadata as structured fields in _data.
  3. The PDF bytes are base64-encoded and appended to truck.imgs as a pseudo-image entry: cam=98 (reserved for permit documents), ext='.pdf', data=<base64 PDF>, name='Permit', event=Permit_Validated.
  4. If invalid, Alert_InvalidPermit is appended; if validation throws, Alert_NoPermit is appended.
  5. Parsed permit summary is also injected into truck.ticket._data so it shows up on the local PDF as ticket-overlay text (permit number, GVW, status).

The wire format fix (commit on tasks/86b9y6jc5 after the durability work): _slurpImage previously hardcoded .jpg as the filename extension regardless of what the source code set on the img dict. Permit PDFs were arriving on the cloud filesystem as PermitValidated_<ts>_98.jpg — PDF bytes inside a .jpg filename. Browsers and PDF viewers reject those as broken JPEGs.

The fix respects the ext field if present, defaults to .jpg for camera-captured scale images that never set ext:

sExt = i.get('ext', '.jpg')
return {
    'filename': "{0}_{1}_{2}{3}".format(e, e.getTimestamp(), i['cam'], sExt),
    'data': i['data']
}

After this fix, permit PDFs land at <FILE_STORE>/scalewatcher/<serial>/<YYYY>/<MM>/<DD>/PermitValidated_<ts>_98.pdf and are addressable as a standard SW_Event image attachment.

The mechanism is general — any future non-image artifact (signed ticket XML, FastWeigh webhook archive, etc.) can attach the same way by setting ext on its img dict. No further plumbing needed.

CloudAPI side: zero changes. transaction.js:742-765 already iterates current.imgs, writes each img.data (base64) to <dirPath>/<img.filename> with no extension-specific logic. Whatever filename arrives is what gets saved.

Portal side (not yet shipped — separate work item):

The portal's gallery (scalewatcher/src/components/transaction_gallery/ SingleTransaction.js) currently renders all event images via <img> tags. For .pdf attachments the browser would show a broken image.

Required change (single branch in the image renderer):

{filename.endsWith('.pdf') ? (
  <embed src={imageUrl} type="application/pdf" style={pdfStyle} />
) : (
  <img src={imageUrl} ... />
)}

Browser-native PDF rendering, no JS library dependency. The "PDF" icon on the event-key legend can also be added so operators see at a glance which events have permit attachments.

Transaction PDF compilation (deferred):

The portal generates per-transaction PDF compilations via @react-pdf/renderer (see cloudapi/scalewatcher/src/utilities/ parseEventData.js). That library doesn't natively embed external PDFs — it works with images. Recommended approach for the permit: include a "Permit Summary" text section (number, GVW, status, expiration — all from the Permit_Validated event's _data) plus a "View Original Permit" link, instead of trying to embed the PDF page itself. Keeps the transaction PDF compact and preserves the original permit PDF as the canonical archive. If embedding becomes a requirement, the compilation engine would need to swap to pdf-lib which supports PDF merging.

  1. shipmentID phase tag is dropped on the wire. RESOLVED 2026-05-23 (commit 9d3d18b43). The setShipmentID calls were dead code — _prepareEvents never serialized the field, and the reuse conflicted with PSCL's legitimate use of Event.shipmentID for shipment grouping. Removed. If/when the cloud portal needs to distinguish tare-phase from loadout-phase events explicitly, that work lands as part of the option-C paired.tare sub-document (item 3 below), not via the field-overload approach.

  2. Load and Wait time computations at the cloud portal will produce wrong numbers for paired transactions. Park for now per 2026-05-23 discussion; revisit when the option-C upload shape lands so phase-tagged events are available for the computation.

  3. The paired.tare sub-document upload shape (option C from the design discussion) is NOT yet implemented. This document describes the as-shipped behavior, which is option A — both scale cycles' events merged into one events list.

  4. Portal-side PDF rendering is the only remaining piece of the permit-to-cloud pipeline — recorder + cloudapi storage are wired (see Permit PDF Delivery section above). Needs a single-branch change in SingleTransaction.js and an optional "PDF" icon in the event legend.

  5. Test harness wiring for test_sw_cloud_measurement.py: the file isn't included in recorder/tests/run_all.py (no suite() function exported, not in the import list). Tests pass standalone (python3 test_sw_cloud_measurement.py).
