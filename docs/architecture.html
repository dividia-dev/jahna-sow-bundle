1a. Scale Watcher feature stack (added 2026-05-23)

Scale Watcher is a layered product. Each layer is independently togglable per PosType19 row. This came from the David Smith reframing on 2026-05-23 once the FastWeigh integration made it clear "APEX vs FastWeigh vs CommandCloud" is an input-channel axis, not a behavior switch — every downstream layer is provider-agnostic.

Layer Toggle field Provider-dependent Notes
0. Input channel PosType19.sTicketSource (enum: apex, fastweigh, commandcloud) drives UI section visibility only wire-side provider tag on each ticket is authoritative for runtime; declared field is sanity-checked at receive time
1. Local recording always-on no every scale saves tickets + images locally for network-share lookup
2. Cloud upload PosType19.fCloudUpload (per-row) gated by system-level Misc.fCloudSw master no promoted from system-only to per-row so multi-scale sites can opt some scales out
3. Paired tare/loadout PosType19.fPairTareLoadout (per-row) + sMatchInbound/sTareField/sTareValue no works for any provider; merges tare + loadout into one cloud transaction
4a. Axle measurement PosType19.fMeasurement + sMeasurementSource (per-row) no promoted from GB['measurement-source'] global to per-row so 2-scale sites can tag tare-A vs loadout-B independently; GB var retained as fallback for legacy single-scale sites
4b. Permit validation PosType19.fPermitValidation (per-row) no promoted from GB['permit-source'] global; GB var retained as fallback

Provider-agnostic above layer 0. All four upper layers work identically regardless of which input channel feeds layer 1. The CloudTicket adapter (recorder thread_ticket_reader.py:84) is the abstraction primitive that makes this work — APEX XML and cloud JSON tickets become interchangeable to every downstream caller.

Sanity check at receive. thread_truck_transaction.py:_saveTicket compares the wire provider tag against PosType19.sTicketSource. The apex-declared / cloud-arriving case is silently allowed (legacy migration path); other mismatches log a PROVIDER MISMATCH errMsg with both values so support can find the misconfiguration.

Migration: 20260523-sw-feature-stack.py adds five PosType19 columns (sTicketSource, fCloudUpload, fMeasurement, sMeasurementSource, fPermitValidation) and creates TicketLog — a generic per-receive archive that replaces ApexLog for non-APEX providers (whose string-shaped uniqueids don't fit ApexLog.bUniqueID BIGINT). ApexLog stays in place for legacy APEX flow until that path migrates separately.

0a. DataBridge scope — much smaller than initially assumed

[VERIFIED — customer call 2026-05-21]: "DataBridge scope — Only other function beyond AxlEye: plugin that renames jpg+xml with ticket# and timestamp. No other plugins/integrations to worry about."

This is a single point of leverage for quoting. Replacing DataBridge means replacing AxlEye (which we already do) plus one tiny file-rename behavior — not replacing a meaningful orchestration layer. The Tier 2 / Tier 3 quote scope should reflect this.

0b. Operational rules (must be honored in code or workflow)

  1. Measurement is INBOUND/tare ONLY. Loader tablet needs max weight BEFORE loading. No re-measurement on outbound.
  2. Drop axle MUST be deployed for inbound measurement. If the driver doesn't drop axle, they don't get extra weight — the truck physically can't position for valid axle measurement. Mention in operator UI, validate visually in the measurement screenshot if possible.
  3. Permit shown by driver every truck, every time. No exceptions, even returning drivers. Permits are per-permittee, not per-vehicle.
  4. Mini-wheelers (dump trucks) skip measurement. Operator clicks tire count: 3 tires → 64,500 lbs cap · 4 tires → 70,000 lbs cap. Photo only, no axle measurement. UI affordance needed.
  5. Trucks without OW permit fall back to standard bridge formula based on FL DOT table + truck length. Most trucks DO have OW permits.

0c. The customer's real operational flow (from customer call)

This is the "tare scale sequence" — everything happens while the truck is on the inbound scale:

  1. Truck pulls onto tare scale
  2. Driver holds up permit to QR camera at scale-house window
  3. System reads QR, validates against FDOT public URL, captures permit PDF
  4. Measurement camera captures full side view; operator/system measures axles (with drop axle deployed)
  5. Files dropped to shared folder (this is what AxlEye + DataBridge does today; we replace this end-to-end)
  6. Operator in scale-house UI: select hauler / truck / customer / contract, click "Get Weight", click "Accept"
  7. Truck marked InYard with predetermined max weight → loader tablets see it
  8. Truck drives to loading circle, gets loaded against that max-weight target
  9. Returns to outbound scale; operator selects from InYard, adjusts weight with clam-shell bucket, clicks "Complete" → ticket closed, PDF printed (no pictures on PDF), some customers also get emailed ticket

The loader-tablet feed at step 7 is the open Tier-2/3 scope question — see § 10 item 4.

10. Cross-cutting assumptions still on faith
