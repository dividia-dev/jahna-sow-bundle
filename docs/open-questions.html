If it's FastWeigh, we may need to push measurement data into FastWeigh via their API so the tablets can display it.)
  • How does the max weight data get to the loader tablets today? Through DataBridge's "InYard" status? Through FastWeigh? A shared database? A network share?
  • What other data do the tablets show? The customer mentioned hauler, truck#, and product. Where does each of those fields come from?

Why this matters: If we can't feed the loader tablets, the operator has no max weight target and the core value prop of inbound measurement is broken. This determines whether we need a FastWeigh write-back API, a DataBridge-compatible shared folder drop, or our own tablet display solution.

1.0a QR Permit Reader — Mobile App vs Mounted Camera

The existing AxlEye setup uses a dedicated Uniview IPC3635SB-ADZK-I0 camera mounted at the scale house window for drivers to hold up their permit. This is separate from the measurement camera (IPC2324SR5-ADZK-G).

[UPDATED 2026-05-25] Chosen approach: mounted NVR camera (auto-detect, hands-off). Driver pulls up, holds permit to the camera, and the system does everything from there — frame-level QR decode → snapshot → permit lookup → attach to transaction. No operator click, no driver phone interaction.

[CLARIFIED 2026-05-25] Two dedicated NVR-connected cameras, no sharing. AxlEye is removed entirely from our deployment — we own the entire QR + measurement pipeline. The hardware layout mirrors what Jahna runs today (which AxlEye drives today) but our software, not theirs, does every step:

  • Close-range driver-window camera — QR-enabled, our code reads QR codes off this stream and emits events.
  • Long-range side-view camera — feeds the spot-decoder for axle measurement.

The spot-decoder display (the screen the operator looks at while measuring) can later show picture-in-picture or status overlays — e.g., "QR scan attempted", "permit OK", "permit invalid" — to give the operator visibility into the QR pipeline without having to look at the dview live grid. This is a follow-up polish, not a v1 requirement.

The mobile-app option remains a longer-term differentiator but is not the v1 path:

  • v1 customers (Jahna) have an established workflow built around a window-mounted camera

  • Self-scan adoption is uncertain; the mounted path works for every driver day one

  • The mounted approach is also the cleanest one-for-one swap for the existing rig

  • Decide: mobile app or reuse mounted camera? → Mounted camera, auto-detect.

  • Decide: dedicated QR camera, or shared with measurement camera? → Two separate dedicated cameras, both on the NVR. No PTZ swap, no shared stream.

1.0c Measurement Automation Roadmap — long-tail vision

Captured 2026-05-25 during the §1.0b decision call. The print-time fallback (§1.0b, shipped this branch) is the safety net, not the destination. Beyond it lies a roadmap toward full automation of the side-view measurement:

  • Phase 2 (THIS BRANCH). Operator-driven measurement is still the happy path. If they forget to press save, the recorder grabs the spot-decoder JPEG + QR-cam frame at TicketReceived as visual evidence; emits Alert_NoMeasurement / Alert_NoQrScan so the cloud surfaces the gap.

  • Phase 3 (NEXT). Add a measure-now endpoint to the spot-decoder (recorder posts it at TicketReceived as part of the fallback). The spot-decoder runs its existing axle-distance algorithm against the current frame without waiting for the operator's "save" click. Likely a 1-2 day spot-decoder PR. After Phase 3 lands, the §1.0b fallback upgrades from "image only" to "image + best-effort measurement" with zero recorder changes.

  • Phase 4 (NEXT-NEXT). AI on the spot-decoder side-view auto-locates the first + last axle when the truck is stationary in frame and auto-counts wheels/axles. Replaces the manual click-the-two-lines pattern entirely. The operator's role becomes review/override rather than primary capture. Same fallback path keeps working — it'd just succeed much more often.

  • Phase 5 (FARTHEST). Front-of-scale camera reads axles as the truck rolls onto the scale. Fully automatic, no stationary-frame requirement, no operator interaction at all.

The Phase 2 fallback we're building stays as the residual safety net through every later phase. Even after Phase 5, a sensor failure or unusual truck geometry might leave the AI without a measurement — the fallback catches those.

1.0b Measurement Fallback — Snapshot + Best-Effort On Save Miss

Today's measurement flow on the NVR side (verified nvr/recorder/src/scalewatcher/measurement_client.py + thread_truck_transaction.py:765-775):

  • measurement_client.fetch_measurement() polls the spot-decoder over HTTP at the moment a transaction enters _handleFinishedState
  • If the spot-decoder returns a measurement and it isn't stale, the transaction gets the measurement attached + a MeasurementRecorded event
  • If no measurement is available, the transaction gets an Alert_NoMeasurement event and proceeds with no axle data

Gap: there is no "grab whatever you can at ticket-print" fallback. If the operator didn't save a measurement on the spot-decoder before the ticket prints, the transaction goes out with no measurement and no snapshot. For a bridge-formula audit trail that is too lossy — at minimum the truck-side image should land on the transaction.

The cheapest fallback path uses what's already there:

Step What Where
1 measurement_client already exposes _fetch_screenshot() returning the latest spot-decoder JPEG measurement_client.py:130
2 When fetch_measurement() returns no measurement, additionally call _fetch_screenshot() and attach the JPEG to the transaction as a spot-decoder-cam image new code in _handleFinishedState
3 Emit a MeasurementBestEffort event (or extend Alert_NoMeasurement with the snapshot reference) so the cloud transaction record carries the image trivial, follows existing event patterns

Open questions (DECIDED 2026-05-25):

  • Hook point? → TicketReceived event handler in _handleProcessingState. At this point the truck is still on the scale (it hasn't received a cFinishTransaction / Scale_TruckLeaving yet), so the measurement-cam still has the full truck in frame and the QR-cam still has whatever was held up. _handleFinishedState (the previous default) fires after bTicketPrintDelay + ScaleTruckLeaving — too late, truck may already be moving off the pad. Shipped: snapshot at TicketReceived, consume in _handleFinishedState if real events didn't fire.
  • Force spot-decoder measurement at print time? → No, not in this branch. Image-only fallback for v1 (uses the existing _fetch_screenshot() endpoint, no spot-decoder change). The force-measure endpoint becomes Phase 3 of the §1.0c roadmap.
  • QR-permit camera print-time snapshot? → Yes, always grab. If the QR producer didn't decode anything during the transaction AND permit validation is configured on the POS, the recorder emits Alert_NoQrScan and attaches the last frame the producer saw as cam-97 image. Gives the operator visual evidence of whether a permit was held up but missed by the decoder.

5. Research Findings Summary (for reference)

5.1 Florida Permit Weight Logic

Without permit:

  • Max weight = bridge formula result (W = 500((LN/(N-1)) + 12N + 36))
  • Capped at 80,000 lbs (or 70,000 for single-unit dump trucks per 316.535(6))
  • Per-axle: 20,000 lbs single, bridge formula for groups
  • Tables include 10% scale tolerance already

With nondivisible blanket permit:

  • Max GVW from FDOT Weight Restrictions Chart (based on axle count + outer bridge + routing map)
  • Per-axle limits from permit face (e.g., SINGLE: 22K, TANDEM: 44K, TRIDEM: 60K)
  • BOTH GVW and per-axle limits must be satisfied
  • Outer bridge must meet minimum on permit (or permit is VOID)
  • Tire contact pressure must not exceed permit max (600 lbs/in)

With divisible load permit (sand/aggregate):

  • Max GVW = bridge formula result x 1.1 (10% over standard)
  • Per-axle limits are NOT raised (standard 316.535 limits apply)
  • Non-interstate only
  • Does NOT apply to single-unit dump trucks (they stay at 70K cap)

5.2 Bridge Formula Implementation Status

  • cv-axle-controller implementation is CORRECT (verified against F.S. 316.535)
  • All 60 tests pass
  • Missing: per-axle-group limits, permit override logic (planned for v2)
