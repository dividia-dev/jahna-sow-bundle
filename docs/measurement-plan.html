Measurement Integration Plan

CORRECTED 2026-05-22, extended 2026-06-02: This doc was authored 2026-05-18 with an assumption that measurement happens on the OUTBOUND/loaded ticket. Customer call 2026-05-21 (see memory project_jahna_email_response_20260521.md §1) confirmed the opposite — measurement is INBOUND/tare ONLY so the loader tablet can show predetermined max weight BEFORE loading. Strikethrough on individual lines below preserves diff history; the corrected direction governs.

ALSO CORRECTED 2026-06-02: Any references below to FastWeigh transactionGuid as the correlation/anchor source are superseded. Tare↔loadout correlation keys on a configurable PosType19 tare-ticket field (default truck_id), provider-agnostic across Apex XML and FastWeigh JSON. Existing mechanism = fPairTareLoadout + sMatchInbound/sTareField/sTareValue. See paired-mode-flow.md. "How we deliver our measurement data to FastWeigh" is a separate parked concern, not in current test scope.

Architecture Overview

Four systems, one flow. Measurement + permit checks happen on the INBOUND/tare ticket (loader tablet uses the resulting max weight BEFORE loading; no re-measurement on outbound).

FastWeigh (SaaS) ──webhook──> CloudAPI (cloud) ──websocket──> NVR (on-site) ──LAN──> Spot Decoder (on-site)
                                  ^                              |
                                  └──────── upload txn ──────────┘

System Roles

System Location Role
FastWeigh Cloud SaaS Ticketing. Creates tare/loaded tickets. Sends webhooks via Svix.
CloudAPI Cloud server Backend. Receives webhooks, receives NVR uploads, stores everything, serves portal.
NVR (dview) On-site appliance Records video, manages cameras, runs POS event logic. Persistent WS to CloudAPI.
Spot Decoder On-site Linux box HDMI overlay on live RTSP. Operator measures axles with keyboard/controller. Saves JSON.

Loaded Inbound/Tare Ticket Flow (Step by Step)

Heading kept with strikethrough for diff clarity; flow is INBOUND.

  1. FastWeigh fires ticket.created webhook to POST /sw/webhooks/fastweigh
  2. CloudAPI verifies Svix signature, deduplicates, passes raw ticket JSON through, pushes ticket_event over WebSocket to NVR
  3. NVR receives ticket_event, starts transaction (records video, captures images)
  4. Operator measures truck on spot-decoder — places lines on axles, sets count, hits save. Writes /tmp/measurement-latest.json
  5. NVR fetches measurement from spot-decoder — via HTTP GET /?stream=0&measurement=status on port 88
  6. NVR uploads complete transaction to CloudAPI — images, video, events, measurement data bundled together
  7. CloudAPI stores measurement, runs permit validation (weight-dependent checks), creates alerts on failures
  8. Portal displays everything — ticket, images, video, measurement overlay, permit status

Permit Validation Split

Check Where Why
Axle spacing / bridge formula geometry Spot Decoder (at save time) Operator gets instant feedback on HDMI
Weight-dependent checks (gross vs permit limit) CloudAPI (at transaction landing) Gross weight comes from FastWeigh ticket, not available on spot-decoder

The original Phase 5 plan (OCR + dedicated permit table + cloudside bridge-formula port) is obsolete. Actual implementation:

Spot Decoder: Hotplug + Multi-Device Input (TODO)

The spot-decoder (spot-decoder/src/input.c) currently scans /dev/input/event0..31 once at startup in input_init(), opening a single FD per device type (keyboard, IR, joystick). Two problems:

1. No hotplug support

Devices plugged in after boot are never discovered. Unplug handling works (failed read() closes the FD), but nothing re-scans. This affects:

  • USB keyboard/numpad — must be plugged in before spot-decoder starts
  • HDMI display — hotplug detection needed for display connect/disconnect
  • Arduino controller — same issue

Fix: Add libudev (libudev.h) for enumeration and hotplug monitoring. Use udev_monitor_new_from_netlink("udev") to get a monitor FD, add it to the existing select() call in input_wait(), and on add/remove actions re-enumerate matching input devices.

2. Single FD per device type (multi-keyboard collision)

The Arduino Leonardo measurement controller (VID:PID 2341:8036, serial HIDPC) registers as a USB HID keyboard — same type as a standard USB numpad. Both must be connected simultaneously (Leonardo for dual-joystick axle control, numpad for numeric input). With single-slot-per-type, only whichever appears on the lower /dev/input/event number gets opened.

Fix: Change from single FDs (fdkb, fdjs) to arrays per device type. Match the Arduino controller by its unique product name instead of generic "Keyboard" keyword.

3. Arduino Leonardo firmware identity

The Leonardo's USB descriptor strings are firmware-configurable. Rename from "Arduino LLC Arduino Leonardo" to a professional product name for device matching and appearance.

PlatformIO build flags:

build_flags =
  -DUSB_MANUFACTURER='"Dividia"'
  -DUSB_PRODUCT='"DT-ATC200"'

This changes what shows up in lsusb, dmesg, and the EVIOCGNAME ioctl that spot-decoder uses. Then find_input_device("DT-ATC200", ...) becomes a unique match — no collision with any other keyboard.

Arduino Leonardo USB interfaces:

  • Interface 0: CDC ACM Control (class 2) — serial port
  • Interface 1: CDC Data (class 10) — serial data
  • Interface 2: HID (class 3) — Usage Page 1, Usage 6 = Keyboard

Key files

  • spot-decoder/src/input.c — scan, init, select loop
  • spot-decoder/src/input.h — structs/defines (EVENT_FILES_SCAN_MAX=32, input_t holds the FDs)
  • spot-decoder/src/main.c:320 — where input_t lives as a stack var, passed to HTTP thread
  • spot-decoder/src/conf.c — measurement config section (extend for per-device config)

QR Code Reading from Camera — Where Does It Live? (TODO — DESIGN)

Captured 2026-05-23 (David): before-I-forget item. Test camera expected at the office Tuesday 2026-05-26 (Uniview, model TBD pending what Jahna has — likely matches their IPC3635SB-ADZK-I0 permit-window cam).

The question: When a permit QR code is captured by a fixed camera at the scale house (driver holds the permit up to the window — AxlEye-replacement path, NOT the §1.0a mobile-app path), where does the decode happen?

Two candidate hosts:

  1. Spot Decoder (on-site C process). Already has RTSP frame access for the measurement overlay and runs on the same appliance. Pros: one box for the whole inbound flow; QR result can be saved alongside the measurement JSON. Cons: pulls a QR/zbar dependency into the C code; spot-decoder is currently single-stream per appliance, would need to read a 2nd camera stream.
  2. NVR (recorder / mpengine). Already pulls all camera streams and has the Python POS event loop. Pros: no new stream reader; can push the decoded GUID directly into the active transaction the same way the existing permit_client.py consumes a GUID today. Cons: another decode pipeline in the recorder, may compete for CPU with motion / encode.

Today: permit_client.py takes a permit GUID and pulls the public FDOT PDF (pas.fdot.gov/Public/ViewPermit/{guid}). It does NOT scan a QR — the GUID is currently sourced from either the FastWeigh ticket field (untested against real payload) or operator-entered (also untested). AxlEye does QR scanning today; we don't, so this is a new capability to add.

Next session — concrete steps (Tuesday 2026-05-26 onward):

  1. Confirm the test camera model when it lands at the office. Spec what resolution + frame rate the QR camera streams; sanity-check that QR readability is achievable at the mount distance Jahna uses.
  2. Prototype both: spot-decoder C with libzbar OR Python recorder qr_client.py polling a frame via mpengine. Decide on perf + complexity.
  3. Wire the decoded GUID into the existing permit_client.fetch_permit(guid) pipeline. No changes needed downstream — cloud upload already handles Permit_Validated events.

Cross-link: This is the camera-based alternative to the mobile-app QR scan in open-questions.md §1.0a. Both can coexist; either is sufficient.
