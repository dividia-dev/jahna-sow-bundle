Critical operational rules (from customer call 2026-05-21)

  1. Measurement is INBOUND/tare ONLY. Loader tablet sees max weight BEFORE loading. No re-measurement on outbound.
  2. Drop axle MUST be deployed for inbound measurement. If driver doesn't drop axle, they effectively don't get extra weight (the truck physically can't position for valid axle measurement).
  3. Permits shown by driver every truck, every time. No exceptions — even returning drivers. Permits are per-permittee (carrier), not per-vehicle. Drivers haul for multiple carriers in a day.
  4. Mini-wheelers (dump trucks) skip measurement. Operator clicks tire count: 3 tires = 64,500 lbs cap, 4 tires = 70,000 lbs cap. Photo only, no axle measurement.
  5. Trucks without OW permit fall back to standard bridge formula based on FL DOT table + truck length. Most trucks DO have OW permits.

Mission

Replace AxlEye + DataBridge with our own pipeline so Jahna can measure axle spacing on inbound trucks, validate FDOT overweight permits, ingest FastWeigh tickets, and surface everything in the ScaleWatcher portal. Every truck in, every truck out, with the permit PDF as the audit artifact.

Operating rules for cross-repo work

  1. /checkpoint from cloudapi only for Jahna work. Other repos' checkpoint buckets are siloed and lead to fragmentation.
  2. Update cross-repo-status.md at every session end — that doc is the source of truth for "where are we right now."
  3. Label facts: [VERIFIED file:line] vs [ASSUMED]. If a claim about another repo isn't verified, mark it. Don't move it to VERIFIED without reading code in that repo.
  4. Customer-facing artifacts (emails, permit PDFs, photos) → cloudapi/docs/jahna-fastweigh/customer/ (create when needed).
  5. Memory entries in any repo's user-level memory dir point HERE; they don't duplicate this content.
