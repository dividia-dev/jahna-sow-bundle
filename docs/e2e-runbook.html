E2E test runbook

Jahna FastWeigh — E2E Runbook

Purpose. Reproducible, layer-by-layer verification of the paired-mode + standalone ticket flow across recorder → cloudapi → staging RDS → portal UI. Run this before any merge that touches event names, transaction lifecycle, ticket parsing, paired-mode logic, or the cloudapi transaction.js handler.

Why this exists. The bytes-hit-the-staging-RDS bar is not enough — a row without sTruck or with no surrounding ScaleTruckEntering/QR/Measurement events is a row that LOOKS right in raw SQL and BROKEN in the portal. This runbook walks every layer so we catch the gap.

Companion docs:

  • paired-mode-flow.md — config switches + recorder paths
  • transaction-lifecycle-deep-dive.md — every event in the lifecycle
  • cloud-upload-payload-shapes.md — exact JSON shapes
  • fastweigh-integration.md — webhook signature flow

2.5. The full local-test stack (READ THIS FIRST)

Why this section exists: without all four pieces lined up, "I sent tickets to staging" is a half-truth. My 2026-06-03 session hit this: sent tickets to my local cloudapi which DID write rows to cloudapi_staging RDS, but the deployed staging cloudapi + staging portal don't have the new TareTicketReceived handler/renderer — so the staging portal showed bare rows without tare-half rendering. Anyone running the runbook needs to know they have to bring up local cloudapi + local frontend together to see the full picture.

The four stack pieces

Layer Where it runs What runs it What has the new code
1. VM4001 recorder VM4001 (192.168.0.141) systemd recorder.service rsync from this Mac via setup-once.sh
2. cloudapi backend Mac mini docker container cloudapi-app-1 (port 8081) docker compose in cloudapi repo bind-mounts cloudapi/app/ → /usr/src/app, nodemon auto-reload on file change
3. scalewatcher React frontend Mac mini docker container scalewatcher-frontend-1 (port 3000) docker compose up frontend in scalewatcher repo bind-mounts scalewatcher/ (current branch source)
4. cloudapi_staging RDS AWS us-east-1 managed persists rows — DB schema is shared, NOT versioned per branch

Both the cloudapi container and the scalewatcher container use bind mounts, so editing source on the Mac auto-reloads (cloudapi via nodemon, React via webpack hot-reload). This is the developer-loop guarantee — no rebuild step needed for source edits.

Network plumbing

VM4001 recorder needs to upload to MY cloudapi (not production api.cloud.dividia.net). setup-once.sh patches VM4001's recorder/src/globals.py:

GB[ 'cloud-api' ] = 'https://api.cloud.dividia.net'
GB[ 'cloud-ws' ]  = 'wss://api.cloud.dividia.net:443'

A .bak-e2e-<timestamp> backup is saved; teardown.sh restores it.

React frontend needs to reach my cloudapi (not production). In scalewatcher/package.json:

"proxy": "http://host.docker.internal:8081"

Already set — when the frontend container makes API calls to relative paths (/sw/transactionboarddata?serial=4001&...), webpack-dev-server proxies them to the Mac host's port 8081 (= cloudapi-app-1). No config change needed.

Mocking the scale lifecycle (ScaleTruckEntering/On/Leaving/Left)

Real Moxa scales aren't required. test_scripts/scale_simulator.py listens on a TCP port and emits weight strings in the recorder's scale protocol; the recorder's thread_scale_reader connects to it exactly as it would a real scale and emits Scale_TruckEntering, Scale_TruckOn(weight), Scale_TruckLeaving, Scale_TruckLeft based on weight transitions.

PosScale config on VM4001:

  • bID=11, sIP=127.0.0.2, bPort=2101, bType=7 (Toledo).

So scale_simulator runs on VM4001 itself, binds 0.0.0.0:2101, recorder thread_scale_reader connects to 127.0.0.2:2101.

The wrapper scripts/e2e/lib/scale-sim.sh handles:

  • start — start scale_simulator on VM4001 in background
  • enter <weight> — ramp weight 0 → <weight> (truck enters scale)
  • stable — hold at current weight
  • leave — ramp weight → 0 (truck leaves scale)
  • stop — kill scale_simulator

A full happy-pair scenario looks like:

./lib/scale-sim.sh start
./lib/scale-sim.sh enter 42000    # tare truck enters, weight ramps to 42000
sleep 8                            # wait for ScaleTruckEntering + ScaleTruckOn(42000)
./lib/inject-apex.sh tare --truck T-X --uniqueid $TS --gross 42000
sleep 2
./lib/scale-sim.sh leave           # tare truck leaves
sleep 8                            # wait for ScaleTruckLeaving + ScaleTruckLeft

./lib/scale-sim.sh enter 79700    # loadout truck enters
sleep 8
./lib/inject-apex.sh loadout --ticket 1001 --truck T-X --uniqueid $TS2 --gross 79700
sleep 2
./lib/scale-sim.sh leave
sleep 8

./lib/scale-sim.sh stop

Expected events on the upload payload after the full sequence (per scenario S1):

ScaleTruckEntering
ScaleTruckOn (42000)
TareTicketReceived       ← was TicketReceived, renamed at cloud serialize boundary
MeasurementRecorded
ScaleTruckLeft
ScaleTruckEntering
ScaleTruckOn (79700)
TicketReceived
ScaleTruckLeft

The portal Table.js timeline now shows red scale-truck icons at the scale lifecycle points + the navy ticket icon (or hotpink tare icon) at the ticket points.

Why scale_simulator (and not real hardware)?

  • Real scale = Moxa serial converter + physical load cell + scale indicator hardware. We don't need this for code testing.
  • scale_simulator IS the same scale-protocol output a real scale produces. thread_scale_reader can't tell the difference.
  • Lets us drive any weight curve programmatically — repeatable tests.

5. The 6 scenarios (matrix)

Each scenario is one full transaction shape driven end-to-end. The shape column shows the events the recorder must emit (in order) for the test to be true-to-life.

Paired mode ON (Misc('system','paired-tare-loadout')='on')

Scenario 1 — Happy pair (Apex)

Goal: prove the canonical paired flow works with full event surround.

Tare half (Scale 1):

  1. ScaleTruckEntering ← drive via scale weight crossing entering threshold
  2. ScaleTruckOn ← scale stabilized
  3. QrRead ← QR camera decoded driver permit
  4. MeasurementRecorded ← spot-decoder posted axle/distance
  5. TicketReceived ← Apex XML drop with <ticket>0</ticket><truck>T-S1-001</truck>... ↪ at cloud boundary, renamed to TareTicketReceived because _tare=True
  6. ScaleTruckLeft ← truck rolled off

Loadout half (Scale 1 or 2): 7. ScaleTruckEntering 8. ScaleTruckOn 9. QrRead 10. TicketReceived ← Apex XML drop with <ticket>1001</ticket><truck>T-S1-001</truck>... (same <truck> value as tare → matched via sMatchInbound='truck') 11. ScaleTruckLeft

Expected outcome on staging RDS:

  • 1 row in SW_Transaction, with:
    • bTicket = 1001 (loadout)
    • sTruck = 'T-S1-001' (loadout's <truck> value)
    • sCorrelationId non-NULL
    • sTag = 'truck'
  • ~10-11 rows in SW_Event for this bTransaction, including:
    • 2× ScaleTruckEntering, 2× ScaleTruckOn, 2× ScaleTruckLeft
    • 2× QrRead
    • 1× MeasurementRecorded (tare side only — per project_jahna_save_on_tare_side)
    • 1× TareTicketReceived (with tare:true in sData)
    • 1× TicketReceived (loadout)

Expected outcome in portal:

  • 1 row showing T-S1-001 as the truck name (not ???)
  • Click row → events list shows tare half dim/distinct from loadout half
  • Tare ticket icon visually distinct from loadout (color, not just opacity — see §8 portal color decision)
  • All event icons render: scale, qr/lpr, ticket (×2), measurement, alert (if any)

Scenario 2 — Orphan tare expiry (paired ON)

Goal: prove orphan-tare sweep fires AlertTareWithoutLoadout and uploads the tare even though no loadout arrived.

Tare half only:

  1. ScaleTruckEntering
  2. ScaleTruckOn
  3. QrRead
  4. MeasurementRecorded
  5. TicketReceived ← Apex XML <ticket>0</ticket><truck>T-S2-002</truck>...
  6. ScaleTruckLeft

Wait bPairTimeout + 1 min (61 min with testbed config). During this wait, the periodic expiry sweep in thread_truck_transaction._expireInboundCache detects the stale tare and:

  • Tags every event in truck.cloudEvents with _tare=True
  • Fires AlertTareWithoutLoadout event onto the truck
  • Calls cloudAdd → upload as orphan transaction

Expected outcome on staging RDS:

  • 1 row in SW_Transaction:
    • bTicket = NULL (per TareTicketReceived handler — does NOT set bTicket)
    • sTruck = NULL (same reason — does NOT set sTruck)
    • sCorrelationId non-NULL (one-off)
  • SW_Event rows include:
    • TareTicketReceived (carries the tare ticket # in event.bTicket, not in SW_Transaction.bTicket)
    • AlertTareWithoutLoadout
    • all the surrounding tare-half events tagged tare:true

Expected outcome in portal:

  • Row appears with truck name = '???' (because SW_Transaction.sTruck is NULL — this is the CORRECT behavior for an orphan tare, NOT the same bug as scenarios 5/6 below)
  • AlertTareWithoutLoadout icon (goldenrod exclamation) visible
  • Click row → tare event has tare flag/color
  • Searchable by tare ticket number via SW_Event.bTicket — verify with Ticket Search

Scenario 3 — Loadout without tare (paired ON)

Goal: prove loadout arriving with no cached tare fires AlertLoadoutWithoutTare and uploads standalone.

Loadout half only (no prior tare):

  1. ScaleTruckEntering
  2. ScaleTruckOn
  3. QrRead
  4. TicketReceived ← Apex XML <ticket>3001</ticket><truck>T-S3-003</truck>... ↪ paired logic looks for cached tare by truck='T-S3-003' → MISS → fires AlertLoadoutWithoutTare inline
  5. ScaleTruckLeft

Expected outcome on staging RDS:

  • 1 row in SW_Transaction:
    • bTicket = 3001 (loadout TicketReceived sets it normally)
    • sTruck = 'T-S3-003'
  • SW_Event rows:
    • TicketReceived (NOT renamed — this is loadout-side, _tare never set)
    • AlertLoadoutWithoutTare
    • all surrounding events

Expected outcome in portal:

  • Row with truck name T-S3-003
  • Goldenrod AlertLoadoutWithoutTare icon
  • No tare-half visible (because there isn't one)

Scenario 4 — Interleaved paired pairs

Goal: prove sMatchInbound='truck' correctly correlates two simultaneously-cached tares with their respective loadouts.

Sequence:

  1. Tare for truck A — <ticket>0</ticket><truck>T-S4-A</truck>
  2. Tare for truck B — <ticket>0</ticket><truck>T-S4-B</truck>
  3. Loadout for truck A — <ticket>4001</ticket><truck>T-S4-A</truck>
  4. Loadout for truck B — <ticket>4002</ticket><truck>T-S4-B</truck>

Each tare and loadout is preceded by ScaleTruckEntering etc. as in S1.

Expected outcome on staging RDS:

  • 2 rows in SW_Transaction:
    • Row A: bTicket=4001, sTruck='T-S4-A'
    • Row B: bTicket=4002, sTruck='T-S4-B'
  • No cross-contamination: A's events are not on B's row and vice versa

Expected outcome in portal:

  • Two distinct rows for T-S4-A and T-S4-B, each with their tare + loadout events

9. Common gotchas

9.1 sMeasurementSource IP staleness

Symptom: AlertNoMeasurement events on every scenario. Cause: PosType19.sMeasurementSource pinned to old IP (e.g., 0.187 instead of 0.69). Fix: UPDATE PosType19 SET sMeasurementSource='192.168.0.69:88' WHERE bID=1; then restart recorder.

9.2 Paired toggle source confusion

Symptom: setting PosType19.fPairTareLoadout has no effect. Cause: as of 2026-06-02 (commit 336dadc28), the toggle moved to Misc('system','paired-tare-loadout'). The per-POS column is dropped. Fix: write to Misc table, not PosType19. See project_globalize_paired_toggle.

9.3 Off-string boolean regression

Symptom: paired mode behaves as ON even when Misc.sValue='off'. Cause: bool('off') is True in Python (any non-empty string is truthy). Pre-2026-06-02 bug. Fix: ensured by commit 977147f14 (explicit string compare). Verify with grep -n 'paired-tare-loadout' recorder/src — should compare against 'on'.

9.4 2x cloud upload (multi-scale NVRs)

Symptom: 2 SW_Transaction rows per ticket on multi-scale NVRs. Cause: thread_cloud_websocket fans every ticket event to all POS threads. Pre-fix, each fan-out fresh-sCorrelationId'd and defeated dedup. Fix: ensured by commit cd1ae227c (port routing in addEvent). Verify with grep -n '_port' recorder/src/scalewatcher/thread_truck_transaction.py.

9.5 dview test-api session modal

If using dview test-api for orchestration, a stale session modal blocks input. Per memory feedback_dview_modal_blocks_input, modals are invisible to default screenshot — use ?mode=robot or probe via dismiss-modal {"text":"Cancel"}.

9.6 0.144 build host broken

The canonical deploy path (build on 0.144 → VirtualBox VMs) is broken. Workaround: direct rsync to VM4001 — vboxsf writes through. See memory project_dot144_broken_state.
