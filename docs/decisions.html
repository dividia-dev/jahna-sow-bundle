Architecture Decision Records — Jahna FastWeigh Integration

One ADR per non-trivial design choice. Date the decision. Note what was considered. Note what would force a revisit. When a decision is reversed, append a new ADR; don't rewrite history.

ADR-001 — Project hub lives in cloudapi/docs/jahna-fastweigh/

Status: Accepted Date: 2026-05-22

Context. The Jahna FastWeigh feature spans 4 repos. gstack checkpoints, user-level memory, and CLAUDE.md are all scoped per repo directory. Two parallel Claude sessions (in nvr and cloudapi) had divergent context with no shared source of truth, leading to conflicting assumptions and wasted work.

Decision. Pick cloudapi as the project hub. Master docs live at cloudapi/docs/jahna-fastweigh/. Every other repo gets a one-line pointer doc and a memory entry routing back here.

Why cloudapi over alternatives:

  • vs nvr (most code): cloudapi already had docs/jahna-project-open-items.md and 3 related design docs. Customer-facing artifacts (emails, permit PDFs) belong with REST/webhook code, which is in cloudapi.
  • vs a sibling workspace (~/Desktop/dev_projects/jahna-fastweigh/): would not be in git. If cloudapi is the repo most active on the project, anchoring docs there keeps them version-controlled with related code and visible to teammates.

What forces a revisit. If cloudapi gets archived or the project's center of gravity shifts entirely to nvr, move the hub.

ADR-002 — No dedicated SW_Measurement table; ride on SW_Event with summary on SW_Transaction

Status: Accepted Date: ~2026-05-20 (predates this ADR file — recorded retrospectively from measurement-integration-plan.md § 4)

Context. A naive approach would model measurement as its own first-class entity with a SW_Measurement table joined to SW_Transaction via a foreign key. Pros: cleaner schema, easier portal queries, native joins.

Decision. Measurement rides as a MeasurementRecorded event row inside SW_Event (full payload in sData JSON). Summary fields bAxleCount (int) and fBridgePass (tinyint) live directly on SW_Transaction so the portal can list/filter without parsing event JSON.

Why:

  • Measurement is one of many events that already accumulate on a transaction — separating it makes the schema asymmetric without need.
  • NVR already batches everything (ticket, scale events, measurement, permit, alerts) into a single upload. Treating measurement as just another event row in that batch matches the existing pipeline.
  • Summary on SW_Transaction covers all known portal filter/list use cases without joins.

What forces a revisit. If portal develops query patterns that need first-class measurement entities (e.g., "show all measurements regardless of transaction status," historical measurement aggregations independent of trucks), promote to its own table. [VERIFIED migration: cloudapi/app/migrations/20260520000000-add-sw-transaction-measurement-columns.js]

ADR-003 — Measurement UI moves from cv-axle-controller browser app to spot-decoder C

Status: Accepted Date: 2026-05-19 (recorded in cv-axle-controller/.claude.../memory/project_spot_decoder_decision.md)

Context. Original implementation was a browser/Node.js app in cv-axle-controller. Operators reported that browser focus issues made it unreliable in a multi-window scale-house environment — keystrokes intended for measurement went to the wrong window.

Decision. Production measurement UI runs on the spot-decoder device (C, Linux framebuffer, direct USB HID input). The browser app and Arduino firmware stay in cv-axle-controller as the reference implementation and hardware test harness.

Why C over staying in browser:

  • Direct framebuffer rendering — no window manager, no focus to lose.
  • Direct /dev/input/event* USB HID — no browser keystroke interception.
  • Already integrated with the existing spot-decoder video pipeline (FFmpeg, overlay rendering, HTTP API).

What forces a revisit. If a use case demands operator interaction outside the scale house (e.g., reviewing a recorded measurement on a tablet), we'd reintroduce a web/native UI for that path — but spot-decoder remains primary for live measurement.

Implementation status. nvr/spot-decoder/src/measurement.c (838 lines), 111/111 bridge-formula tests passing. Awaiting hardware test (open-questions.md / PROGRESS.md "Blocked" section).

ADR-004 — NVR is the source of truth for transaction boundaries; CloudAPI is a write-through ledger

Status: Accepted (retroactively documented — implicit in current code) Date: 2026-05-22 (writeup), implementation dates back to original recorder design

Context. A truck's data spans multiple events: ticket arrival, scale events, axle measurement, permit validation, alerts, images. Two reasonable models:

  • A — NVR holds an open transaction across the truck flow; uploads once at finish. NVR's TruckTransaction object accumulates everything, then sends a single batch.
  • B — Cloud holds an open transaction; NVR posts events individually as they happen; cloud merges them by transaction id.

Decision. Model A. NVR's TruckTransaction is the transaction; CloudAPI just inserts one row per upload.

Why A over B:

  • NVR has the operational context (which physical truck, which scale, which lane) that lets it group events correctly even if FastWeigh ticket and scale event arrive seconds apart.
  • Single upload simplifies retry semantics — NVR queues a JSON file, retries on failure, deletes on 200. No partial transaction state to reconcile.
  • Cloud doesn't need a state machine; just an insert path. Far simpler to reason about.

Implementation:

  • NVR STATE_PROCESSING → STATE_FINISHED then cloudAdd(scale_id, events, imgs) once [VERIFIED nvr/recorder/src/scalewatcher/thread_truck_transaction.py:617-691, :827]
  • CloudAPI POST /sw/ does one INSERT + one UPDATE + one batch INSERT events [VERIFIED cloudapi/app/controllers/scalewatcher/transaction.js:651-671, :885-894, :898-917]

What forces a revisit. If we ever need cloud-side merging (e.g., two scale houses reporting on the same truck independently), we'd need transaction-id reconciliation at cloud. Not on the roadmap.

ADR-005 — FastWeigh webhook ingestion: raw JSON pass-through, no normalization

Status: Accepted Date: 2026-05-18 (recorded in fastweigh-integration.md)

Context. Existing scale integrations (APEX, Command Alkon) have their own ticket schemas. The naive approach to a new vendor is to normalize their fields to our internal model.

Decision. Pass the FastWeigh webhook ticket through to NVR as raw JSON. Strip only routing fields (serial, port, type) at the cloud edge; tag provider:'fastweigh'. The NVR-side CloudTicket adapter wraps the dict to expose the .rgsOverlay, .bTicket, .bUniqueID, etc. interface that the rest of NVR expects from a Ticket object.

Why:

  • FastWeigh's payload shape is documented (Svix-delivered JSON) but field names may drift over time. A normalization layer becomes a maintenance burden every time FastWeigh adds a field we care about.
  • The NVR side already supports adapter-pattern ticket sources (APEX Ticket, CloudTicket). Adding a new provider is changing one adapter, not a translation layer.
  • Reduces cloud-side complexity to almost nothing — cloud is mostly a proxy + persistence layer.

Implementation. cloudapi/app/controllers/scalewatcher/fastweigh.js:25-40 (prepareTicket) and :56-74 (pushTicketToNvr). Adapter on NVR side: recorder/src/scalewatcher/thread_cloud_websocket.py:82-87 wrap-in-CloudTicket.

What forces a revisit. If we end up integrating 3+ webhook providers (FastWeigh, FastWeigh competitor, in-house), the duplication across NVR adapters may justify a thin normalization layer. Not yet.

ADR-006 — No dedicated SW_Permit table; permits ride as events, same as ADR-002 for measurement

Status: Accepted (retroactively documented — implicit in current code as of 2026-05-21) Date: 2026-05-22 (writeup), decision predates this file

Context. Original design (open-questions.md, pre-2026-05-21) considered modeling permits as a first-class entity with SW_Permit joined to SW_Transaction via FK, and considered whether the relationship was per-truck or per-permittee. Original migration even added a bPermit INT FK column on SW_Transaction in anticipation.

Decision. Same shape as ADR-002 (measurement). Permits ride as Permit_Validated event rows in SW_Event (parsed fields in sData JSON), and the permit PDF rides as a transaction image attachment. No dedicated SW_Permit table. The pre-existing bPermit column on SW_Transaction is a vestige with no FK target and can be dropped in a cleanup migration whenever — not blocking.

Why:

  • Same arguments as ADR-002: the data already flows through the per-truck batch upload; carving out a separate entity makes the schema asymmetric without need.
  • Customer confirmed (2026-05-21) that permits are per-permittee (drivers switch trailers across the day), so a per-truck FK would have been wrong anyway. A per-permittee entity would force us to invent a "permittee" identity layer that adds maintenance burden and doesn't unlock new portal queries.
  • The Permit_Validated event already carries everything: permit number, expiration, validity, parsed fields, errors, GUID. The PDF is right there as an image. Portal queries that need "all permits validated in date range X" or "all transactions with invalid permits" can run against SW_Event with type=Permit_Validated.

What forces a revisit. If a portal use case emerges that requires permit-as-first-class (e.g., "show me every transaction that used permit 12345 regardless of carrier or date"), we'd add a thin index table — not a full entity. Note: same trigger logic as ADR-002 for measurement.

Cite: nvr/recorder/src/scalewatcher/permit_client.py (parser), thread_truck_transaction.py:779 (event), :788 (PDF image), cloudapi/db/init.sql:636 (the vestige column).

ADR-007 — FDOT PAS access uses public URL from permit QR code, no credentials

Status: Accepted Date: 2026-05-21 (implementation), 2026-05-22 (writeup)

Context. Original assumption (open-questions.md §1.2 as originally written) was that we'd need authenticated FDOT PAS credentials to validate permits, mirroring how AxlEye accesses PAS. Concern was that without auth we couldn't reach permit detail. Open item flagged: "Get Jahna's PAS credentials or a sub-account for us."

Decision. Skip credentials entirely. The QR code physically printed on every FDOT overweight permit decodes to a public PAS URL: https://pas.fdot.gov/Public/ViewPermit/{guid}. That page is publicly accessible. The clean PDF is at https://pas.fdot.gov/Public.aspx/ShowPermitDocument/{guid}. We extract the GUID from the QR text, download the PDF binary, parse it, and we're done. No login, no scraping the ASP.NET UI, no credentials.

Why:

  • Public endpoints are stable; ASP.NET pages drift. The PDF format is regulator-controlled and changes far less often than the FDOT web UI.
  • No credential management, no rotation, no compliance overhead, no Jahna-specific account setup.
  • Faster path to first deployment — no pending blocker on customer-side provisioning.

What forces a revisit. If FDOT ever locks the /Public/ShowPermitDocument endpoint behind auth, we'd need to re-evaluate. They could also rate-limit; not currently a problem at one validation per inbound truck.

Cite: nvr/recorder/src/scalewatcher/permit_client.py:7-9, :44, :85-100, :136. Verified against real permit ecda589c-7aaa-4c7b-9771-122133e533fd per memory project_permit_validation.md.

ADR-008 — Measurement is gated to tare-side (inbound) only

Status: Accepted (retroactively documented — implementation already in production code) Date: 2026-05-24 (writeup), implementation predates per feat/scalewatcher-feature-stack

Context. Customer (Jahna, 2026-05-21 email) confirmed measurement happens on the inbound/tare leg of a paired transaction so the loader tablet can see the truck's max allowed weight before loading. Bridge formula is computed from axle geometry, which is the same whether the truck is empty or loaded — measuring on inbound is the natural choice. Measuring on outbound too would either double the work or risk overwriting the inbound measurement with a stale/garbage one.

Decision. The recorder fetches measurement (and validates permit) only on the tare side of a paired transaction. Loadout tickets do not trigger measurement. Discriminator is fIsTareForMeasurement, computed from paired-mode config plus the _isTareTicket() classifier.

Why:

  • Matches the customer's actual workflow — inbound is when the operator runs the measurement, and the result must be ready when the loader tablet picks up the truck.
  • One source of truth per transaction. No reconciliation between two measurements of the same truck.
  • Cleaner alert semantics: Alert_NoMeasurement / Alert_StaleMeasurement only meaningful on the side that should have produced one.

What forces a revisit. If a site ever runs only outbound scales (no separate tare house, only loaded weighing), or if a customer asks for after-loading axle-spread verification, we'd need a per-site override. Not on the roadmap.

Cite: nvr/recorder/src/scalewatcher/thread_truck_transaction.py:723-775 — fIsTareForMeasurement gate around fetch_measurement() and Measurement_Recorded event creation. Alert events Alert_NoMeasurement / Alert_StaleMeasurement only fire from the tare branch.
