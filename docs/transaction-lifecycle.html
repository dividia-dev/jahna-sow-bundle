NVR Transaction Lifecycle — Deep Dive

Naming note (2026-05-23): the feature was initially called "bundled-mode" during design and first implementation. Renamed to "paired-mode" on 2026-05-23 — identifiers, columns, hub doc, and prose updated in one atomic refactor (nvr branch tasks/86b9y6jc5). A few historical quotes below preserve the original "bundle/bundled" wording for context.

Scope: NVR-internal transaction model only. Read of the actual current code in ~/Desktop/dev_projects/nvr/recorder/src/scalewatcher/*, recorder/src/thread_main.py, and dview/src/net/dividia/dview/backend/config/PosType19.java. Every claim has a [VERIFIED file:line] cite.

Question this answers: Can the NVR start a single transaction at the tare scale, hold it open, capture loadout-side events when the truck reaches the outbound scale, and emit one paired upload — for both the combined-scale and separated-scale physical layouts?

Short answer up top: Not natively. The current state machine is one-transaction-per-POS-instance, no cross-scale correlation, and treats each tare cycle and each loadout cycle as separate independent transactions. The pairing model is achievable but requires concrete code additions on the NVR side. Both Tier 1 (minimum surgery) and Tier 3 (clean architecture) options are sketched at the bottom.

Author / refresh: 2026-05-22, model-correction banner added 2026-06-02 Status: Analysis only — no code changes. No commitment implied.

MODEL CORRECTION 2026-06-02: This doc's "Tier 1 / Tier 3" framing and any references to transactionGuid / ticketPending as the correlation source are superseded. Current model: tare↔loadout correlation keys on a configurable PosType19 tare-ticket field (default truck_id), provider-agnostic across Apex XML and FastWeigh JSON. Existing mechanism = fPairTareLoadout + sMatchInbound/sTareField/sTareValue. See paired-mode-flow.md for the live mechanics and architecture.md § 1a for the provider-agnostic stack. The "how we hand FastWeigh our measurement data" question is parked / not in current test scope.

1. Executive summary

Capability Current state Pairing-friendly?
Hold a transaction open while accumulating events YES, STATE_PROCESSING does exactly this YES, this is the foundation
Configurable start / finish triggers per direction YES, sStartTransaction(Inbound) and sFinishTransaction(Inbound) in PosType19 config YES, the hooks exist

Auto-finish on max-open-time exceeded YES — bHungTransTimeout (default 60 min, max 60 min in current UI) YES, but the 60-min UI cap may need lifting

Cross-POS-instance correlation (tare scale POS ↔ loadout scale POS) NO. Each POSType19 instance has its own thread, own truck_list, no shared state NO, this is the second blocker for separated-scale layouts
Cloud-ticket routing to specific POS instance PARTIAL. CloudAPI sends port in the ticket_event WebSocket message, but the NVR-side handler ignores it and broadcasts to all POSType19 instances NO, needs per-port targeting on NVR side
Per-event direction tagging (so cloud can tell tare event from loadout event in the same batch) NO. All accumulated events are dumped into one cloudEvents[] list. The bundle doesn't tell you which scale fired which event without inference NO, needs an extra field on each event or a sub-section in the upload payload

Bottom line: the NVR has the primitives (open state, event accumulation, configurable triggers, hung-timeout, state pickling) but lacks the composition (multi-phase transactions, cross-POS linking, ticket-to-NVR-port routing) to do the bundle natively. The composition is buildable on top of what's there; no rewrite needed. § 6 and § 7 sketch the two paths.

3. The state machine

Three states, with explicit transition rules. Driven by the run() loop in ThreadTruckTransaction (:489-555).

3a. The main loop (:504-547)

Every iteration:

  1. Get next event from the in-queue (qEvent, max depth 10). qEvent.get(True, 0.1) so empty queues poll at 10Hz. [:1249-1258]
  2. Process hung transactions — call _processHungTransactions() (forces finish on any truck whose timer is past bHungTransTimeout). [:508, 2026-2039]
  3. If no event: continue (so hung-check runs every poll regardless of event traffic).
  4. Get truck for this event via _getTruckForEvent(event) (returns the single truck for POSType19; creates one if none exists, with default direction "Outbound"). [:512, 1983-2012]
  5. Maybe delay finish via _shouldDelayFinishTransaction(truck, event) (CRH-specific workaround: scale finishes before APEX prints ticket, so wait bTicketPrintDelay seconds for a late ticket). [:517, 561-590]
  6. Overlay event on live video (_overlayLiveEvent) and force-record on all cameras (_forceVideoRecording). [:521-524]
  7. Track truck-on-scale flag for Scale_TruckEntering/Scale_TruckLeaving. [:527-530]
  8. Process scale events (capture tare/max weight from Scale_TruckOn/Scale_TruckLeaving, fire tare alert if exceeded). [:535, 847-867]
  9. State machine dispatch based on truck.state. [:538-545]

3b. STATE_WAITING (_handleWaitingState, :595-632)

Trigger Transition Notes
Scale_TruckOnAtStartup stays WAITING, sets fSkipTruck=True Truck was already on scale when recorder booted — skip all reporting for this visit
cls == cInboundDirectionTrigger stays WAITING, sets direction to "Inbound" For POSType19, cInboundDirectionTrigger is None (line 336, 363) — dead code path. Direction-switch never fires
cls == truck.cStartTransaction (and not a no-op Valve_Opened-without-truck-on-scale edge case) → PROCESSING Normal start
cls == TICKET_RECEIVED_EVENT → PROCESSING (force) If a ticket arrives without a start event, accept it
Error events (Scale_LostConnection, Scale_RestoredConnection, Scale_DataFail, Valve_DataFail) → PROCESSING (force, fForcedByError=True) Bubble errors to cloud ASAP

3c. STATE_PROCESSING (_handleProcessingState, :634-691)

This is the event accumulator. Every event passes through here while in this state.

Action When Cite
Start timer if not started Always on first event of this transaction :646-647
Grab images for the event class If we haven't grabbed for this event class yet :650-657
truck.addEvent(event) (append to cloudEvents) Always (unless filtered as duplicate Valve_DataFail) :670-671, 120-125
_saveTicket(truck, ticket_or_dict) if cls == TICKET_RECEIVED_EVENT On ticket arrival :674-675, 1535-1545
Force-finish (return STATE_FINISHED) (1) duplicate start event without seeing finish; (2) second ticket with different number; (3) ticket arrives with no truck on scale (when scale is configured); (4) the configured cFinishTransaction arrives; (5) fForcedByError is set :637-643, 662-667, 677-685, 687-689

3d. STATE_FINISHED (_handleFinishedState, :693-841)

This is where upload happens and where measurement + permit are fetched.

  1. Minimum-duration filter — if getDuration() < bMinTransDuration AND no ticket AND not forced by error → CLEAR truck and return to WAITING without uploading. ("bouncing scale" protection.) [:696-702]
  2. Grab remaining images for any cam:event pair not yet captured. [:705, 1396-1487]
  3. Process theft-deterrent reports (alerts on overweight, tare contamination, gross weight mismatch, manual ticket, waste ticket). [:708, 1077-1230]
  4. Fetch measurement from spot-decoder if measurementClient is configured. Adds Measurement_Recorded event + cam 99 screenshot image. Fires Alert_StaleMeasurement or Alert_NoMeasurement as appropriate. [:711-762]
  5. Fetch and validate FDOT permit if permitClient is configured AND _findPermitQrText finds a pas.fdot.gov URL in the event stream or ticket data. Adds Permit_Validated event + cam 98 PDF image. Fires Alert_InvalidPermit or Alert_NoPermit. [:768-824, 1355-1394]
  6. Upload — self.cloudAdd(self.bID, truck.cloudEvents, truck.imgs) — single fire-and-forget dispatcher callback to ThreadCloudSW.addEvent. [:827]
  7. Save ticket files locally (inbound or standard, with overlay rendering). [:830-834, 879-1075]
  8. Clear truck (delete image files, reset state) → return STATE_WAITING. [:837, 2014-2024]

3e. Hung-transaction force-finish (_processHungTransactions, :2026-2039)

Runs every loop iteration. Iterates truck_list[sName]. For any truck whose checkTimerTimeout(self.bHungTransTimeout) returns True, directly calls _handleFinishedState(truck) — bypassing the normal state-machine dispatch. This means the timeout path emits whatever events have been accumulated, exactly as the normal finish would.

The timer starts at the first event of STATE_PROCESSING (:646-647), not at transaction creation. A truck sitting in WAITING does not count toward the timeout.

3f. State persistence (saveState/loadState, :1907-1970)

The full truck_list[sName], plus any queued-but-unprocessed events, plus fTruckOnScale and fSkipTruck, are pickled to /tmp/rda/recorder_thread_{bID}_state.pkl on shutdown. loadState() restores everything on startup. Survives recorder restart but not necessarily a full machine reboot (depends on tmpdir survival).

Implication for the pairing question: a transaction held in PROCESSING across a recorder restart survives. Useful — if the recorder crashes mid-cycle, the tare-side events aren't lost.

Critical observation about the cloud ticket path:

  • The CloudAPI pushTicketToNvr (in cloudapi/app/controllers/scalewatcher/fastweigh.js:56-74) embeds a port field in the data payload alongside ticket/number/provider.
  • The NVR on_message handler in thread_cloud_websocket.py:82-87 does not read port. It just fires Ticket_Received(number, ticket_data_dict) and the ticketPushEvent dispatcher fans out to every subscribed POSType19 instance.
  • Every POS19 instance on the recorder sees every cloud ticket. Whichever is in PROCESSING (or transitions to PROCESSING in response) will save the ticket via _saveTicket.
  • For a multi-POS setup (separated tare + loadout scales = two POS19 instances), ticket routing is broadcast, not targeted. Both POS instances would attempt to claim the ticket. The second-ticket-different-number force-finish path (:662-667) would partially shield, but the design is racy.

5. Scale event emission

The scale reader (thread_scale_reader.py) runs as a separate thread, one per scale device. It polls the scale serial/TCP feed and emits the four scale event classes.

Event When emitted Cite
Scale_TruckEntering() On motion detected + weight rising past bTruckEnterMinWeight. Also force-emitted if Scale_TruckOn arrives without a prior entering event (recovery path) :594, 630
Scale_TruckOn(weight) Once weight stabilizes above bTareMinimum and we don't yet have a tare. The first stable reading is the tare weight :598-603
Scale_TruckLeaving(tareWeight, maxWeight) When weight drops below bTareMinimum while we have a bMaxWeight recorded (truck is off). Also fired on disconnect-then-stable-near-zero (recovery) :614, 655
Scale_TruckOnAtStartup() On startup if a truck is already on the scale :448
Scale_LostConnection, Scale_RestoredConnection, Scale_DataFail On error transitions (other lines in same file)

Each scale reader has its own state. A truck on the tare scale fires the four-event sequence (Entering → On → Leaving → optional second On if it lingers). Drives away, comes back loaded — that's ANOTHER full sequence on the SAME scale (single-scale layout) or the SAME sequence on a DIFFERENT scale reader (separated-scale layout).

The reader has no concept of "this is the same truck as five minutes ago." It just emits events as it sees them.

5b. The existing APEX tare/loadout linking mechanism — ApexInboundCache is already in production for POSType19

Critical finding I underweighted in earlier sections. POSType19 (Scale Watcher) already has a tare-loadout linking mechanism that's been in production for APEX customers for years. The mechanism operates on the LOCAL NVR file/DB layer — it does NOT pair the cloud upload, but it DOES merge tare-side images into the loadout-side combined PDF locally. The pairing work in §§ 10-11 below is an extension of this existing pattern, not a from-scratch invention.

5b.1 The two ticket types

APEX itself emits two distinct ticket types that flow through this NVR as Ticket_Received events with different bTicket values:

Ticket type Discriminator Path in _handleFinishedState
Tare/inbound ticket bTicket == '0' (literally the string zero) calls _saveInboundTicket → adds to ApexInboundCache
Loadout/standard ticket bTicket != '0' (a real numeric ticket number) calls _saveStandardTicket → pulls matching inbound from cache, merges into local PDF

When a tare-side transaction reaches STATE_FINISHED with bTicket == '0':

  1. Each truck.imgs[] entry is copied to GB['inbounddir'] with filename pattern Inbound.{POS_id}.{timestamp}.{cam_num}i.jpg :896
  2. Tare-ticket overlay text is burned onto each image :905-911
  3. self.oInboundCache.add(ticket.oXML, bTimestamp, rgsJPG) :918 writes:

    • If a prior cache entry for the same matching-key exists, purges it first :46-49
  4. Note: cloudAdd has ALREADY fired at line 827 BEFORE this code runs, so the cloud has already seen the tare-only transaction. The cache is for local PDF assembly only.

  3. ApexInboundCache.get(oTicket) does the matching apex_inbound_cache.py:56-94:

5b.4 Expiration (_expireInboundCache :869-877 + ApexInboundCache.expire :96-128)

  • Runs every 600 seconds (INBOUND_EXPIRE_SLEEP) from the main loop via _handleWaitingState :598
  • For any cached inbound older than self._bTimeout seconds (from bTimeoutInbound * 60), generates a REPORT_INBOUND exception report ("this tare never got matched to a loadout, here are the orphan images")
  • Cleans up the MariaDB row + JPG files

5b.7 Finalized design — paired-cloud-upload extension

Status: Design approved 2026-05-22. Documented here as the locked design. Implementation pending quote acceptance.

5b.7.1 Three config flags on PosType19

Flag Type Default Purpose
fPairTareLoadout boolean false Master switch. When false, code path is exactly today's behavior (per-cycle cloud upload, ApexInboundCache for local PDF combining). When true, paired mode: suppress tare-side cloudAdd, cache full TruckTransaction via extended ApexInboundCache, merge into loadout upload, fire ONE cloudAdd with combined events + images + measurement + permit.
sTareField string "bTicket" Which ticket field identifies a tare. APEX convention is bTicket. FastWeigh would be whatever they expose (type, weighing_step, is_tare, etc.) — confirmed via FastWeigh sales call.
sTareValue string "0" The value that field holds on a tare ticket. APEX prints bTicket == "0" for tares. FastWeigh might emit "tare" or "true" or "1".

Why field+value instead of an eval()-able expression: operator-friendly in dview UI, no security risk from eval, no Python-side parsing of operator config.

5b.7.5 Code surgery in thread_truck_transaction.py

Three checkpoints where the new flags are consulted, all in _handleFinishedState (:693):

def _handleFinishedState(self, truck):
    # ... existing: minimum-duration, image grab, theft, measurement, permit ...

    if self.fPairTareLoadout:
        is_tare = self._isTareTicket(truck)        # uses sTareField/sTareValue

        if is_tare:
            # Paired tare: suppress cloudAdd, cache full state
            self._cachePairedTareTruck(truck)     # pickle into ApexInboundLog.sPairedState
            self._clearTruck(truck)
            return STATE_WAITING

        # Paired loadout: look up held tare, merge if found
        held = self._getHeldPairedTare(truck.ticket)
        if held is not None:
            truck.merge(held)                      # existing TruckTransaction.merge :127
        else:
            truck.addEvent(Alert_LoadoutWithoutTare())

    # Always: fire cloudAdd (now with merged data if paired+loadout)
    self.cloudAdd(self.bID, truck.cloudEvents, truck.imgs)

    if truck.hasTicket():
        if self.fInboundEnable and self._isTareTicket(truck) and not self.fPairTareLoadout:
            self._saveInboundTicket(truck)         # legacy mode only
        else:
            self._saveStandardTicket(truck)        # builds local PDF (merged imgs in paired mode)

5b.7.7 Event-phase tagging in the merged upload — REVISED

So the cloud can distinguish tare-side from loadout-side scale events in the paired payload:

  • Previously recommended (REJECTED 2026-05-23): repurpose Event.shipmentID. We tried it (commit 57c21f04d) and removed it (commit 9d3d18b43). Two reasons it didn't work as planned:
    • thread_cloud_sw._prepareEvents actually only serializes timestamp/name/data/imgs into the upload JSON; shipmentID was NOT included, so the cloud never saw the tag. The "zero cloud schema change" framing assumed serialization that wasn't there.
    • Event.shipmentID is Ryan's PSCL infrastructure (used by _getTruckForEvent to group events into truck transactions by shipment). Overloading it for a paired-mode phase tag put a non-PSCL value into a field PSCL POS code reads.
  • Current state: events ride into the cloud unmodified. Phase is implicit from each event's own ticket data (a Ticket_Received whose number==sTareValue is the tare; anything else is the loadout). Cloud-side code that needs to render tare/loadout distinction can already do it from the per-event payload.
  • If a future feature genuinely needs an explicit per-event phase tag, add a NEW field on Event (e.g. phase) — don't overload shipmentID a second time. That work lands as part of the option-C paired.tare sub-document upload shape, not as a one-line shim.
  • DO NOT invent new event classes like TareTicketReceived and LoadoutTicketReceived. The events themselves are identical (a Scale_TruckOn in tare phase has the same shape as Scale_TruckOn in loadout phase). The phase is metadata about WHEN, not a new event type.

6. The two physical scale layouts

6a. Single combined scale (one physical scale, truck crosses twice)

Truck visit:
  09:00:00  Scale_TruckEntering            (tare)
  09:00:08  Scale_TruckOn(42000)           (tare weight = 42K)
  09:00:30  [operator runs AxlEye + permit QR + DataBridge entry]
  09:00:45  Scale_TruckLeaving(42000, 42000)
            [truck drives to loading circle, gets loaded, returns]
  09:18:30  Scale_TruckEntering            (loaded)
  09:18:36  Scale_TruckOn(76200)           (now-loaded weight)
  09:18:55  Scale_TruckLeaving(42000, 76200)  [tare is from this scale's persistent state OR is 0]
            [FastWeigh ticket arrives some time later via webhook]

How the current code handles this:

  • ONE POSType19 instance is configured for this scale.
  • ONE scale reader emits four events for the tare cycle.
  • The TruckTransaction enters PROCESSING on whatever cStartTransaction is configured (default Ticket_Received per :185, 344 if not overridden, or Scale_TruckEntering if explicitly configured).
  • If cStartTransaction = Scale_TruckEntering and cFinishTransaction = Scale_TruckLeaving, the tare cycle becomes a self-contained transaction. It uploads on tare-leaving.
  • Truck leaves, comes back. ANOTHER Scale_TruckEntering arrives. Since the prior transaction is FINISHED (already uploaded and cleared) — a new TruckTransaction is created. Another self-contained transaction uploads on loaded-leaving.
  • Result: TWO separate cloud transactions per truck visit. No pairing.
  • If cStartTransaction = Ticket_Received and cFinishTransaction = Ticket_Received (default), then only the ticket triggers a transaction at all — scale events get captured INTO the transaction only if a ticket arrives while the truck is on the scale.

The current code has no "wait for the SAME truck to come back" logic. It cannot tell that the 09:18 entering event is the same truck as the 09:00 entering event.

6b. Separated tare and loadout scales (different physical scales)

Truck visit:
  09:00:00  [tare scale]   Scale_TruckEntering
  09:00:08  [tare scale]   Scale_TruckOn(42000)
  09:00:45  [tare scale]   Scale_TruckLeaving(42000, 42000)
            [truck drives to loading circle, gets loaded]
  09:18:30  [loadout scale] Scale_TruckEntering
  09:18:36  [loadout scale] Scale_TruckOn(76200)
  09:18:55  [loadout scale] Scale_TruckLeaving(0, 76200)  [no prior tare on this scale]
            [FastWeigh ticket arrives via webhook]

How the current code handles this:

  • TWO POSType19 instances are configured — one per scale, distinct names ("Tare Scale 1", "Loaded Scale 1"), distinct bIDs, distinct ports.
  • Each has its own ThreadTruckTransaction thread, own scale reader, own truck_list, own qEvent.
  • They do not share state. No cross-thread communication. No shared truck object.
  • Tare scale's POS instance emits ONE transaction (tare-side events + maybe alerts).
  • Loadout scale's POS instance emits a SEPARATE transaction (loadout-side events).
  • The FastWeigh ticket, when it arrives, goes to BOTH instances (broadcast — see § 4). Whichever one is in PROCESSING at that moment captures it. With the loadout scale in PROCESSING when the ticket arrives, the loadout transaction gets the ticket.
  • Result: TWO separate cloud transactions, with the ticket attached to one of them (usually the loadout side). No pairing.

In addition: the design doc fastweigh-integration.md (now superseded) proposed linking by LPN — that's the LPR-based correlation, off the table.

6c. Why the current code emits two transactions either way

The state machine is fundamentally one cycle = one transaction. A "cycle" is defined by the configured start event arriving, events accumulating, and the configured finish event arriving. The cStartTransaction/cFinishTransaction config is per-direction (Inbound/Outbound), but POSType19 never switches direction (cInboundDirectionTrigger = None, :336, 363), so it always uses the outbound triggers. Even if the inbound triggers were activated, the state machine can only hold ONE cycle at a time per POS — there's no "phase 1 / phase 2" within a transaction.

7. What this CAN do (without code changes) — the achievable pairing-adjacent things

Given the existing primitives, you CAN do these things today without code surgery:

  1. Capture all scale events + the ticket into one transaction, when the truck is on a single scale and the ticket arrives within bHungTransTimeout. Configure cStartTransaction = Scale_TruckEntering, cFinishTransaction = Ticket_Received. Truck enters → scale events accumulate → ticket arrives → upload fires with everything. If the ticket arrives late, the timer + hung-timeout protect against an infinitely-open transaction.
  2. Capture multi-event sequences within one cycle. Whatever events arrive between start and finish get bundled. Scale + valve + LPR (if used) + measurement + permit all land in one upload.
  3. Wait for a late ticket via bTicketPrintDelay (CRH-specific workaround, :286, 561-590). After the configured finish event, the code waits up to bTicketPrintDelay seconds for a ticket. This is a different mechanism from the bundle, but the primitive of "delay finishing to wait for an event" exists.
  4. Force-finish on errors so transient scale/valve issues don't leave a transaction hung.
  5. Survive a recorder restart mid-transaction (state pickling, § 3f).

What this CANNOT do without code changes: wait for an event from a different POS instance, or for a tare cycle to "pair" with a loadout cycle that happens 5 minutes to several hours later, or attach a "phase" tag to events so the cloud can tell which scale fired which.

8. What CANNOT be done without code changes — the limitations

In order of how blocking each is for the paired-transaction goal:

8a. POSType19 never switches direction at runtime (HARD LIMITATION)

_parseTTconfigPosType19 parses sStartTransactionInbound and sFinishTransactionInbound from config (:342-349) but the only thing that would switch the active direction is cInboundDirectionTrigger — and that is explicitly hardcoded to None for POSType19 (:336, comment at :361-363: "Disabling this for now until we rewrite to configure Type19 TruckTransaction directly"). The inbound config is parsed-but-dead-code.

Every transaction therefore uses the Outbound triggers (set in _setDirection at :1272-1284 and used by default in _getTruckForEvent at :1992, 2007).

The infrastructure for inbound/outbound is present and parsed — just not wired up for POSType19.

8b. POSType19 is single-truck-per-instance (HARD LIMITATION for the held-cycle model)

_getTruckForEvent (:1983-2012) always returns the single truck in truck_list[sName] for POSType19 (since shipmentID is always "0"). If a tare transaction is in PROCESSING and a NEW Scale_TruckEntering arrives, the duplicate-start force-finish path triggers (:637-643) — the current transaction is closed and a new one starts.

This means a tare cycle and a loadout cycle, even on the same scale, cannot coexist as ONE transaction.

PSCL has a multi-truck workaround via shipmentID, in a separate thread (thread_truck_transaction_pscl.py). That code path exists but is PSCL-specific.

8c. Two POS instances have no shared state (HARD LIMITATION for separated scales)

Tare scale POS and loadout scale POS run in separate threads with separate truck_lists. Nothing in the current code joins them. Even if you fixed (8a) and (8b), a tare-scale TruckTransaction and a loadout-scale TruckTransaction are different objects in different threads.

8d. Cloud-ticket routing is broadcast, not per-port targeted (MEDIUM)

thread_cloud_websocket.py:82-87 ignores the port field that cloudapi sends. Every POSType19 sees every ticket. This is workable in the single-POS case but creates ambiguity in the multi-POS case.

CloudAPI side (cloudapi/app/controllers/scalewatcher/fastweigh.js:56-74) already includes the port field — only the NVR side needs to consume it.

8e. No per-event direction/phase tagging (MEDIUM — UPDATED 2026-05-23)

When the upload payload is built (thread_cloud_sw.py:_prepareEvents), every event in cloudEvents[] is serialized identically (timestamp, name, data, imgs). There's no field saying "this scale event came from the tare cycle" vs "this scale event came from the loadout cycle".

The current paired-mode upload contains a merged list of tare-side + loadout-side events. The cloud disambiguates by reading each event's own ticket data — the tare's Ticket_Received has number==sTareValue; the loadout's doesn't. Scale events that don't carry ticket data have to be associated by timestamp + position relative to the ticket events.

An earlier attempt (commits 57c21f04d → 9d3d18b43) used Event.shipmentID as a phase tag. That was removed — see § 5b.7.7 — because the cloud upload never read the field and it conflicted with PSCL's legitimate use.

The clean fix if/when a portal feature needs explicit phase tagging: either a new Event.phase attribute that DOES get serialized, OR the option-C paired.tare sub-document upload shape. Not by overloading shipmentID again.

8f. bHungTransTimeout is capped at 60 minutes in current UI (SOFT)

ApexDialog.java:103: JSpinner(new SpinnerNumberModel(60, 0, 60, 1)) — max value is 60 minutes. The Python side multiplies by 60 to get seconds (thread_truck_transaction.py:292) and has no inherent ceiling. The UI cap is the constraint.

For a paired tare→loadout cycle, typical times might be 15-30 minutes (truck drives to loading circle, loads, returns). 60 minutes is workable for the normal case but tight for any edge (driver delay, multiple-truck-on-yard congestion, equipment failure). Lifting the UI cap is a one-line dview change.

8g. Minor: minimum-duration filter would need a tweak

_handleFinishedState (:696-702) skips upload if getDuration() < bMinTransDuration AND no ticket AND not forced-by-error. For a paired transaction where the tare side might initially have a short duration before the loadout phase begins, this filter would only fire if we never got a ticket and never had errors — likely harmless, but worth keeping in mind.
